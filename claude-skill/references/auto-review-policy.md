# Auto-review policy

When a peer reaches `status: done, integrationStatus: pushed`, the supervisor runs an automated review in the peer's worktree before opening and merging the PR. The review has exactly two gates.

## Gate 1: forbidden-touch diff

```
git -C <worktreePath> diff --name-only origin/<default_branch>..HEAD
```

For each changed file `f`, fail if any `p` in `config.forbidden_paths` matches:
- `f == p` (exact file)
- `f.startswith(p)` (directory prefix; entries should end with `/`)

…unless `current_slice_id` is in `config.forbidden_exception_slices`.

A hit halts the chain with reason `<slice_id>: peer touched forbidden paths: [...]` and Telegrams the offending file list.

**Why the slice exception list exists.** Some roadmaps explicitly schedule a slice whose job is to visually migrate or refactor preserved files. That slice gets listed in `forbidden_exception_slices` so its diff doesn't trip the alarm. Every other slice is held to strict no-touch.

## Gate 2: verification commands

Each entry in `config.verification_commands` runs sequentially in the peer's worktree. All must exit 0.

Default suite for JS/TS repos:

```json
[
  {"label": "npm run lint",      "cmd": ["npm", "run", "lint"]},
  {"label": "tsc --noEmit",      "cmd": ["npx", "tsc", "--noEmit"]},
  {"label": "npm test -- --run", "cmd": ["npm", "test", "--", "--run"]},
  {"label": "npm run build",     "cmd": ["npm", "run", "build"], "timeout_seconds": 900}
]
```

A failure halts with reason `<slice_id>: verification failed at <label>` and Telegrams the failed labels plus the last ~1500 chars of the first failing command's output.

## Both gates green

The supervisor:
1. `gh pr create --base <default_branch> --head <merge_branch> --title <prefix>(<slice>): <slice> --body <auto-generated>`
2. `gh pr merge <pr_url> --<merge_strategy> [--delete-branch]`
3. `codex-peers cleanup <peer_id>` (best-effort; non-fatal if it errors)
4. Telegram: `✅ <slice> auto-reviewed + merged` with PR URL

## What auto-review does NOT check

- **Semantic correctness of the implementation.** That's the peer prompt's job. If your peer prompt is loose, auto-review is loose.
- **Visual regressions.** Build success ≠ visual parity. If parity matters, add a Playwright/visual-regression command to `verification_commands`.
- **Performance.** Add a benchmark step if needed.
- **Security review.** Add `gh pr view --json` checks or a SAST step if needed.
- **The peer's commit messages.** Trusted as written.

## What "halt" actually does

- Sets `state.halted = true`.
- Writes `state.halted_reason = <text>`.
- Sends one Telegram with the reason.
- Returns 0; cron keeps firing but every subsequent tick exits early at the halted check until the user clears it.
- Does **not** revert the peer's branch on origin. Does **not** delete the peer worktree (intentional — leaves evidence for inspection).
- Does **not** open or close any PR (auto-review halt happens before `gh pr create`; merge halt happens after `gh pr create` but before `gh pr merge` if create succeeded).

If the user wants to cancel a halted peer entirely, they run `codex-peers kill <peer_id>` and `codex-peers cleanup <peer_id>` manually, then either skip or re-spawn the slice.

## Tuning

Verification command order matters: put the cheapest first so a failing lint short-circuits before a 60-second build. The supervisor still runs all commands and reports all failures, but exiting at the first failure-tail in the Telegram message saves the user a scroll.

Single-tick budget: aim for total verification time < 4 minutes so two cron ticks don't pile up against the `flock`. If your repo's full suite is slower, raise the cron interval (e.g. `*/10`) to match.
