## Does it autonomously run GSD in a custom Pi harness?

**Yes — this is its core purpose.** [VERIFIED local: `docs/auto-mode.md`, `src/resources/extensions/gsd/auto.ts` (1817 lines)]

- **How autonomy works:** Auto mode (`/gsd auto`) is a **file-driven state machine**. `deriveState(basePath)` reads `.gsd/` (or `.planning/`) — `STATE.md`, `ROADMAP.md`, milestone/slice/task files — to compute the *next unit of work*. The loop: `Research → Plan → Execute (per task) → Complete → Reassess Roadmap → Next Slice → … → Validate Milestone → Complete Milestone`.
- **How it drives Pi:** Two mechanisms. (1) **In-process, event-driven** — the engine runs *inside* pi as an extension; it dispatches a unit by calling `ctx.newSession()` (fresh context window per unit) then `pi.sendMessage({customType:"gsd-dispatch", content: prompt, display:false}, {triggerTurn:true})` (`auto-direct-dispatch.ts`). The loop is not a `while`; it is re-entered by the **`handleAgentEnd`** event — when the LLM's turn ends, `handleAgentEnd` re-reads disk state and dispatches the next unit (`auto.ts`). (2) **Subprocess** — parallel milestone workers spawn separate pi processes: `spawn(process.execPath, [binPath, "--mode", "json", "--print", "/gsd auto"], {env:{GSD_MILESTONE_LOCK, GSD_PARALLEL_WORKER:"1"}})` (`parallel-orchestrator.ts:428`). Pi also exposes `--mode rpc` (JSON over stdin/stdout: `prompt`/`steer`/`follow_up`) and a TS SDK (`createAgentSession`) for embedding [VERIFIED local: `docs/what-is-pi/14-…`].
- **The phase/plan loop uses fresh sessions per unit** with all context pre-inlined (task plan, slice plan, prior summaries, dependency summaries, roadmap excerpt, decisions register) so each unit starts oriented; `KNOWLEDGE.md` gives cross-session memory.
- **How it terminates:** `deriveState` reports `phase === "complete"` / no active incomplete milestone → `stopAuto(ctx, pi, "All work complete")`. Guardrails that also terminate/pause: **stuck detection** (same unit dispatched twice → one deep-diagnostic retry, then stop), **timeout supervision** (soft 20m warn / idle 10m / hard 30m pause), **budget ceiling** pause, **provider-error classification** (rate-limit/server auto-resume; permanent errors pause). **Headless auto-restart** (`gsd headless auto`, exponential backoff 5s→10s→30s, `--max-restarts N`) + crash-recovery-from-lock give true overnight "run until done."

## Architecture

[VERIFIED local: `parallel-orchestrator.ts`, `docs/parallel-orchestration.md`, `auto-worktree.ts`, `worktree-manager.ts`]

- **Coordinator/worker model:** the coordinator = your `/gsd` session; it does eligibility analysis, worker spawning/lifecycle, budget tracking, signal dispatch, merge reconciliation. Each milestone → one worker pi process.
- **Worktrees:** one git worktree per milestone at `.gsd/worktrees/<MID>/` on branch `milestone/<MID>`; squash-merged to main on completion. `git.isolation` ∈ `worktree` (default) / `branch` / `none`.
- **IPC (file-based):** `.gsd/parallel/<MID>.status.json` (worker heartbeats/progress, read by coordinator) and `.gsd/parallel/<MID>.signal.json` (coordinator→worker pause/resume/stop). **Atomic write-temp-then-rename.** Both dirs are gitignored (runtime-only). SIGTERM augments the stop signal.
- **State/checkpointing:** `.gsd/auto.lock` tracks current unit; on crash the next run synthesizes a recovery briefing from every tool call that reached disk and resumes. Per-worktree `.gsd/metrics.json`. `deriveState()` scoped per worker via `GSD_MILESTONE_LOCK`.
- **Concurrency:** `max_workers` 1–4; eligibility = not-complete + `dependsOn` satisfied + file-overlap *warning* (not blocker; conflicts caught at merge). Merge auto-resolves `.gsd/` state conflicts, **halts on code conflicts**.
- **Nested-spawn guard:** `if (process.env.GSD_PARALLEL_WORKER) return` at coordinator entry (`parallel-orchestrator.ts:261`) — a worker cannot start its own parallel session.
- **Stale detection:** dead PID (`process.kill(pid,0)`) or heartbeat > 30s → cleaned by `/gsd doctor --fix`.

## Relationship to GSD's phase/plan model and to delamain's gsdRunner/spawnGsdPhaseBatch

[VERIFIED local: `~/dev/delamain/src/gsdRunner.ts`, `src/mcpServer.ts`, `src/gsdMilestone.ts`, `src/frozen-gate/`, `src/frozen-eligibility/`]

Both operate on the **same GSD spec model** (`.planning/`/`.gsd/`: PROJECT/ROADMAP/STATE, phases). But they are architecturally opposite:

- **delamain `runGsdPhaseBatch`** is an **external supervisor** that drives a *different* harness — **codex, not pi**: `codex exec --cwd <repo> --json -- /gsd-autonomous --only <phaseId>` (dynamic) or `/gsd-execute-phase <phaseId> --no-transition` (frozen). It's a **terminating `for` loop** over `selected_phases[cursor…]`: spawn codex → await exit → `readStateDocument(repo)` → `isPhaseComplete(state, phaseId)` ? advance cursor : `gsd_failed`; exhausted → `gsd_completed`. It drives GSD-the-**skill-system** (the `/gsd-autonomous`, `/gsd-execute-phase` slash-command skills present in this environment), not gsd-pi's compiled engine.
- **gsd-pi `auto`** compiles the loop into TypeScript, drives **pi** sessions in-process (event-driven, not a for-loop), and adds a decade of hardening delamain lacks: crash-recovery/resume, supervisor timeouts, stuck-retry, adaptive reassess/replan, per-unit fresh sessions, budget, parallel worktree orchestration, KNOWLEDGE.md memory, context-pressure wrap-up.
- **What delamain has that gsd-pi does NOT:** a **"frozen contract" gate** (`gateFrozenPhase` → `gsd_running_gate_check` / `gsd_halted_on_gate_failure`, writes a gate-failure artifact and short-circuits the batch) and a **frozen-eligibility classifier** (`classify_frozen_batch` MCP tool). gsd-pi's closest analog is `verification_commands` + the milestone-validation gate + reassess — softer than delamain's hard contract gate.
- **Verdict on the job:** gsd-pi does the **same autonomous-phase-loop job far more completely**, but only for a **pi-native, single-engine** world. delamain is a **multi-harness (codex/cursor/pi) orchestrator** — a superset ambition gsd-pi doesn't serve. `inspect_gsd_milestone` (delamain clones repo, reads `.planning/`, returns per-phase readiness flags) has **no gsd-pi equivalent** as an external tool; gsd-pi's `deriveState` is internal.

## Reuse verdict for delamain

**(a) SP1 workflow engine (autonomous GSD phase loop) — LEARN-FROM (mine patterns), do NOT adopt the code.** gsd-pi's `auto.ts` is the reference implementation of exactly SP1, but it is welded to pi's `ExtensionAPI`/`ctx.newSession()`/`handleAgentEnd` event model and assumes pi is *the* harness. Adopting it wholesale means rebuilding delamain as a pi extension — contradicting delamain's codex/cursor/pi multi-peer design. Instead port the *patterns* (all directly applicable to delamain's subprocess model): disk-state-as-source-of-truth via `deriveState`→next-unit; **fresh session/subprocess per unit** with context pre-inlined; **stuck detection = same-unit-dispatched-twice → one diagnostic retry → halt**; **three-tier timeout supervision**; **crash recovery via lock + tool-call replay**; **provider-error classification (auto-resume vs pause)**; **reassess/replan-after-slice**; **KNOWLEDGE.md incremental memory**; **context-pressure wrap-up signal**; **headless auto-restart with backoff**.

**(b) Fold/replace delamain's `gsdRunner`? No — enhance it, keep its shape.** `runGsdPhaseBatch` already has the correct external-supervisor shape (spawn-per-phase → STATE.md poll → advance/halt) and adds the **frozen-gate + frozen-eligibility** capabilities gsd-pi lacks — keep those; they're delamain's differentiator. The gap is that `gsdRunner` is *primitive*: a bare terminating for-loop with **no crash-recovery/resume, no timeout supervision, no stuck-retry, no reassess, no budget**. Upgrade it by grafting the (a) patterns onto the existing loop rather than swapping in gsd-pi code.

**(c) SP4 Pi cockpit — VENDOR / LEARN-FROM.** gsd-pi ships a real control plane worth harvesting: `gsd --web` (visual control plane), the **workflow visualizer** (progress/deps/metrics/timeline tabs), self-contained **HTML milestone reports** (`.gsd/reports/`, inlined CSS/JS, SVG dependency DAG), and a **TUI dashboard** (`Ctrl+Alt+G`). External cockpit prior art to vendor/study: **`open-gsd/gsd-pi-config`** (Tauri + React GUI over `~/.gsd/preferences.md`) and **`Kile-Thomson/Rokket-GSD`** (VS Code ext: streaming chat + parallel-worker view). If delamain wants to *actually drive pi peers* (it currently drives codex/cursor), the spawn recipe is proven here: `pi/gsd --mode json --print "<cmd>"` for subprocess, `--mode rpc` (JSON `prompt`/`steer`/`follow_up`) or the `createAgentSession` SDK for in-process; and **`gutomec/pi-squad-loader`** shows loading subagents as native Pi SDK subagents.

**Bottom line:** gsd-pi is the mature, single-engine reference for delamain's SP1 loop and SP4 cockpit but is architecturally the wrong thing to *adopt* (pi-extension-coupled, single-harness). Treat it as a pattern donor and a vendoring source for the visual/cockpit layer; keep delamain's own multi-harness `gsdRunner` + frozen-gate core and harden it with gsd-pi's proven auto-mode reliability patterns.
