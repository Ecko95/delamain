# Sandbox tech research (Sandcastle + isolated-vm) — 2026-07-16

## Sandcastle — what it is, architecture, sandbox tech, license, reusability verdict

**What it is [web/docs]:** `@ai-hero/sandcastle` — a TypeScript library by Matt Pocock (AI Hero) for orchestrating **AI coding agents** in isolated sandboxes. GitHub `mattpocock/sandcastle`, ~6.9k stars, 691 forks, 43 releases, **v0.12.0** (June 2026), **MIT** licensed, published to npm. It is *not* a V8/script sandbox runtime and *not* a course — it is a real, actively-maintained agent orchestrator. Tagline: "Orchestrate sandboxed coding agents in TypeScript with `sandcastle.run()`." Note it is **pre-1.0 (0.x)** → API still churning; Pocock has publicly floated (X, June 2026) ripping out Docker to make the sandbox fully pluggable (local or remote), so the provider contract is in flux.

**Architecture / sandbox tech [web/docs]:** git worktrees + pluggable container runtimes. Built-in providers: **Docker** & **Podman** (bind-mount: host worktree mounted into a container, agent writes through the mount), **Vercel Firecracker microVMs** (isolated FS, `copyIn`/`copyFileOut` sync), and no-sandbox (direct host). Custom providers via `createBindMountSandboxProvider()` / `createIsolatedSandboxProvider()`. It does **not** use isolated-vm / node:vm / workerd — its isolation unit is a **container/microVM wrapping a whole coding agent**, plus git branch strategies for merge-back.

**Primitives [web/docs]:** `run()` (one-shot agent + full sandbox lifecycle), `createSandbox()` (warm reusable sandbox), `createWorktree()`, `sandbox.exec()`, `sandbox.resume()`/`fork()` (session continuation/branching); branch strategies `head` / `merge-to-head` / `branch`; structured-output extraction with schema validation; lifecycle hooks (`onWorktreeReady`, `onSandboxReady`); dynamic prompt injection via `` !`shell-cmd` ``. Agent providers: Claude Code, Codex, Pi, Cursor, OpenCode, Copilot.

**Reusability verdict [inference]:** Sandcastle solves a **different layer** than delamain's workflow-script sandbox. It sandboxes the **agents** (the peers `ctx.agent()` spawns) via worktrees + containers/branch-merge — which is essentially **what delamain already does itself** (`spawn(detached)` peers in worktrees, `--sandbox` modes, merge-branch). It does **not** run untrusted LLM-authored TS *control scripts* with an injected capability `ctx`; there is no in-process script-isolation story to borrow. So: **not vendorable for the workflow-script sandbox.** Its value to delamain is (a) **conceptual confirmation** of the "one call → many parallel sandboxed agents" fan-out goal, and (b) a **reference design for the agent-sandbox provider abstraction and branch strategies** — worth reading `createBindMountSandboxProvider` / `createIsolatedSandboxProvider` and the branch-strategy model if delamain ever refactors its peer-sandbox layer to be pluggable. It is a competitor/sibling to delamain's *own* core, not a component for the new engine.

## isolated-vm — current version, maintenance, Node-25 compat, verdict

**Version/maintenance [VERIFIED locally via `npm view`]:** latest **7.0.0** (published 2026-05-31), `legacy` dist-tag 1.7.11. Repo `laverdet/isolated-vm`. Actively maintained and — importantly, a change from its historical source-only reputation — it **now ships prebuilt binaries** (7.0.0 & 6.1.x depend on `node-gyp-build`; 6.0.2 used `prebuild-install`). Install = `node-gyp-build || node-gyp rebuild` (prebuilt first, compile fallback).

**Node-25 compatibility — DECISIVE, [VERIFIED locally on this Node v25.5.0 machine]:**
- Node 25.5.0 reports **ABI/modules = 141**.
- **7.0.0 declares `engines: { node: ">=26.0.0" }`** → explicitly excludes Node 25 (EBADENGINE). Published May 2026, ahead of a released Node 26 — Node 25 falls in a support gap [inference].
- Engines by version [VERIFIED]: 6.0.0 `>=24`, 6.0.2 `>=22`, **6.1.0/6.1.2 `>=22`**, 7.0.0 `>=26`. So **6.1.2 is the newest version whose engines *allow* Node 25.**
- **6.1.2 prebuilds ship only for ABI 127 (Node 22) and ABI 137 (Node 24)** — glibc+musl linux-x64/arm64, darwin-arm64, win32-x64. **No ABI 141 (Node 25) prebuild, no ABI for Node 23/25/26.**
- I actually installed `isolated-vm@6.1.2` on Node 25.5.0: `node-gyp-build` found no matching prebuild → **compiled from source successfully** (exit 0, ~3 min, needs a C++20 toolchain) producing `build/Release/isolated_vm.node`. **But at runtime it SEGFAULTS (exit 139, core dumped, reproducible): `require()` succeeds, then `new ivm.Isolate({memoryLimit})` crashes the process before returning.** Confirmed twice.

**Verdict:** isolated-vm is the **best-fit isolation *model*** for this use case (see below) but is **effectively unusable on Node 25.5.0 today** — 7.0.0 refuses (needs Node 26), 6.1.2 compiles but segfaults on isolate creation. It is only viable if delamain **pins to Node 24 LTS** (ABI 137 prebuild, works) or later moves to **Node 26** with a 7.x build. Async-`ctx` bridging, memory caps, timeouts, and determinism are all first-class (below) — the blocker is purely Node-version coupling of the native addon.

**Async ctx bridge support [web/docs]:** first-class. Expose host capabilities as `new ivm.Callback(async (argJson) => {...}, { async: true })` set on `context.global`; in-isolate code does `const r = JSON.parse(await ctx.agent(JSON.stringify(spec)))` and receives a **real promise** it can `await`, while the host runs the async work (the subprocess spawn) on the default isolate/event loop. Alternative: `reference.applySyncPromise` (blocks the isolate's own thread until a host promise resolves). Only transferables cross the boundary → serialize args/results as JSON. `memoryLimit` per isolate (V8 heap cap) + per-`run`/`eval` `{ timeout }` (wall-clock CPU kill). Determinism is a strength: the isolate global starts **empty** — no `Date`, `Math`, `require`, `process` unless you inject them — so banning `Date.now`/`Math.random` (or injecting deterministic shims) is trivial.

## RECOMMENDATION for delamain's workflow-script sandbox

**Reframe the threat model first.** The workflow *script* is LLM-authored **control flow** whose only side effects are `await ctx.agent()/parallel()/pipeline()/phase()/verify()`. The genuinely dangerous code is the **agent** those calls spawn — which delamain already isolates in worktrees with Codex/Cursor `--sandbox` modes (`peerManager.ts:440 spawn(detached)`, `cursorRunner.ts:108`, `runner.ts:87`; this is Sandcastle's territory). So the *script* sandbox must deliver four things: capability-only access (no fs/shell/net), resource bounds, determinism, and **run on Node 25.5.0 today**.

**Primary v1 choice: Option 4 — child-process executor + capability bridge, with `node:vm` inside the child for the deterministic global.** Rationale:
- **Works natively on Node 25.5.0** — no native addon, no ABI/segfault risk. isolated-vm is off the table on Node 25 (VERIFIED segfault) unless you pin Node 24; don't accept that coupling for v1.
- **Architecturally native to delamain** — it already runs everything as `spawn(…, {detached:true})` children in worktrees; the executor is one more child of the same shape, and the existing `--sandbox` flag stays the *agent's* sandbox (a different layer, not this one).
- **Layering is the boundary, and this is load-bearing:** `node:vm` **alone is not a security boundary** (trivial escape; vm2 is dead) — use it *only* to control the language-level global (inject `ctx`, omit/deterministically shim `Date`/`Math.random`/`crypto`, freeze intrinsics). The **actual** deny-fs/shell/net boundary is the **OS jail around the child**: unprivileged uid, seccomp/landlock, network namespace (or deny-all firewall), read-only FS, cgroup v2. Optionally also AST-validate the script (delamain already has `tree-sitter`/`tree-sitter-typescript` deps) to reject `require`/`import`/`process` before it runs — cheap defense-in-depth.

**Async-ctx-bridge design (`await ctx.agent()` → real host spawn):** the `ctx` injected into the child is a proxy; each method serializes `{id, method:"agent", args}` and writes it to the parent over the child's IPC channel (or a length-framed JSON-RPC line protocol on a dedicated stdio fd). The **delamain parent** receives it, invokes its existing peer machinery (spawn Codex/Cursor peer in a worktree/container, itself sandboxed), and on completion replies `{id, result}` or `{id, error}`; the child correlates by `id` and resolves/rejects the pending promise, so in-sandbox `const res = await ctx.agent(spec)` returns a structured result. `parallel()`/`pipeline()`/`phase()` are just multiple outstanding `id`s the parent fans out and joins. This keeps all real-world capability (spawning, fs, net) in the **parent**, never the sandbox. **Keep this `ctx` interface identical** regardless of executor implementation.

**Bounding resources to exploit the VPS:** one **cgroup v2 slice per script child** (`memory.max` + `memory.high`, `cpu.max` quota/period, `pids.max`), `cpu.weight` for fairness, plus a wall-clock timeout → SIGKILL. Because control scripts are almost always **IO-waiting on `ctx.agent()`**, meter **two independent pools**: heavily **oversubscribe script children** (hundreds — they're cheap, mostly blocked), but **cap agent concurrency** to the real RAM/CPU budget (agents are the heavy tenants). That directly exploits a beefy VPS: many concurrent workflows, bounded real work.

**Ideal vs pragmatic — flagged:** The *ideal* long-term executor is **isolated-vm** (cheapest per-instance in-process isolation, strongest default-deny, trivial determinism, thousands of isolates in one process) — but it is **Node-version-locked and currently broken on Node 25**. Two migration paths, both behind the *same* `ctx` interface: (a) if you can **pin delamain to Node 24 LTS**, adopt isolated-vm now for the massively-parallel fan-out; (b) otherwise stay on Option 4 for Node 25 and swap in an isolated-vm executor when you move to **Node 26 + isolated-vm 7.x**. Do **not** ship isolated-vm on Node 25.5.0 — VERIFIED to compile-then-segfault. **Container/microVM (Option 5 / Sandcastle-style)** is unnecessary weight for single-tenant, your-own-LLM control scripts; reserve it for the *agent* layer or true remote multi-tenant.
